## 1. Mirendil

**Money.** Mirendil was discovered through a16z's June 24 seed announcement. a16z says it is leading the seed round; the Wall Street Journal reports the round is **$200 million** from Andreessen Horowitz, Kleiner Perkins, and Nvidia at a $1 billion valuation. This is a seed-stage but unusually capitalized frontier-lab company. No revenue, customer, product availability, compute budget, or pricing is disclosed.

**Technology.** Mirendil's user is initially an AI researcher or engineer, and later a domain scientist. The user brings a research goal, codebase, models, datasets, evaluation harness, and compute constraints. Mirendil's system should turn that into experiment plans, generate and run code, compare checkpoints, debug failures, interpret results, decide the next run, and package the resulting model or method for review. Humans still need to define safety boundaries, choose what counts as a valid result, inspect surprising outputs, approve expensive compute runs, and decide whether a benchmark improvement transfers to a real scientific or product target.

The disclosed technical thesis is that frontier models can become expert at AI R&D itself. The architecture likely has three layers: specialized models trained on research-and-engineering traces, an agent harness that can operate repos and compute clusters, and an eval infrastructure that can score experiments without leaking future information. The hard part is not a coding agent that runs scripts; it is reliable closed-loop research. A useful system must manage experiment provenance, negative results, reproducibility, hyperparameter and data lineage, checkpoint comparison, distributed training failures, kernel optimization, and budget allocation.

Commodity pieces include open-source models, code agents, experiment trackers, and cloud GPU orchestration. Mirendil's possible moat is the team, proprietary AI-R&D training data, internal evals, and a compounding platform where each research run improves future runs. The validation gaps are large: no public benchmark, no product, no measured compute efficiency, no safety model, and no evidence yet that the system can discover results beyond tightly specified ML tasks.

**Founders.** Behnam Neyshabur has worked on AI for science at Google and Anthropic; Harsh Mehta reportedly built the first version of Anthropic's autoresearch platform; Shayan Salehian worked on ML engineering at xAI; Tara Rezaei is an MIT graduate and olympiad medalist. This is unusually direct founder-market fit for automating AI research.

**Verdict — 8.6/10, Medium confidence.** Mirendil is the most technically ambitious company in this run and has the strongest founder/financing signal. Confidence is capped because almost all product and performance claims remain pre-release.

## 2. Superset

**Money.** Superset has YC seed financing and no disclosed outside round or revenue. It reports usage by YC companies, engineers at OpenAI, Google, Vercel, and major open-source projects. The GitHub repository shows about **12.1K stars**, about 1K forks, and roughly 2,999 commits. These are strong developer-adoption signals, though paid conversion, weekly active users, and retained team deployment are not public.

**Technology.** An engineer opens Superset on a local repo, creates multiple tasks, and launches Claude Code, Codex CLI, Cursor Agent, Gemini CLI, OpenCode, or another terminal-based agent. Superset creates isolated git worktrees so the agents do not overwrite each other, runs each agent in its own workspace, streams terminal state, tracks which tasks need attention, and gives the engineer a diff/review surface. The output is not an automatic merge; the engineer reviews the branch, edits the diff, opens it in an external IDE or terminal, and decides what lands.

The key system design is local orchestration rather than a model wrapper. Superset needs worktree lifecycle management, repo-specific setup scripts, environment variable handling, agent process supervision, terminal multiplexing, notifications, diff viewing, and safe handoff into editors. Its README indicates Electron, React, Tailwind, Bun, Turborepo, Vite, Drizzle, Neon, and tRPC. The product works because it accepts the reality that coding agents are heterogeneous CLI programs and organizes them around git isolation.

The hard engineering is reliability across messy local repos. Agents need dependencies, databases, secrets, generated assets, dev servers, and branch discipline. A broken setup script or leaked env file can waste hours or expose credentials. Superset's defensibility is not the UI alone; it is the workspace model, integrations, setup conventions, community templates, and the habit of making agent work reviewable. Validation gaps include successful task completion rate, security posture, Windows/Linux support, and whether teams pay once GitHub, OpenAI, Anthropic, and Cursor bundle similar orchestration.

**Founders.** Kiet Ho previously co-founded Onlook and worked at Amazon and ServiceNow. Satya Patel was CTO of Untether Labs and worked at Google and Amazon. Avi Peltz previously co-founded Adam and BioGlyph. Prior YC/developer-tool experience is a real advantage.

**Verdict — 8.3/10, High confidence.** Superset is concrete, inspectable, and already adopted by developers. The main question is business durability, not whether the product exists or solves a real workflow problem.

**Sources:** [YC profile and launch, accessed June 24, 2026](https://www.ycombinator.com/companies/superset) · [GitHub repository, accessed June 24, 2026](https://github.com/superset-sh/superset) · [Product site, accessed June 24, 2026](https://superset.sh/)

## 3. Lightsprint

**Money.** Lightsprint has YC seed financing and no disclosed outside round, revenue, or customer count. The founders say they use Lightsprint to build Lightsprint and that it has increased internal engineering velocity. That is useful dogfooding, but not independent commercial validation.

**Technology.** A product manager, designer, founder, or engineer writes a requirement. Lightsprint turns it into a visual plan with tasks, dependencies, and implementation options. The team comments on the plan, changes scope, and approves it. Lightsprint then spins up multiple cloud coding agents: one may build a frontend component, another writes tests, and another changes a dashboard or backend flow. Each result becomes a pull request with a live preview URL, so non-engineers can click through behavior instead of reading code. Engineers still review architecture, risky changes, tests, and merge readiness.

This is a coordination layer around agentic development. It needs requirements decomposition, task dependency graphs, repo context retrieval, branch/workspace isolation, cloud execution, PR previews, review aggregation, and permissions that distinguish harmless copy edits from database migrations or auth logic. The hard problem is not producing code quickly; it is avoiding a team-wide flood of plausible but incoherent changes. Good Lightsprint output should preserve why each task exists, which requirement it satisfies, how it was tested, and what remains unresolved.

Cloud agents, CI, preview environments, and issue trackers are available components. Lightsprint can differentiate if it becomes the shared planning and review state for non-engineering contributors. The risk is that it overlaps with Superset, Replicas, Stage, GitHub, and model-vendor agent clouds. It must prove that team coordination and preview review materially improve shipped quality, not just task volume.

**Founders.** Ben Ong brings AI-investing and product-scaling experience from January Capital, Temasek's venture studio, SEA Group, and data science. Benedict Chan was VP Engineering at Chainlink and CTO at BitGo. Heng Hong Lee worked at Facebook, led engineering at Fazz, and has deep software leadership experience. This is strong operating fit for developer workflow infrastructure.

**Verdict — 8.1/10, Medium-high confidence.** Lightsprint has a clear wedge: make coding agents team-visible and reviewable. The missing proof is retained customer use, accepted PR rate, regression rate, and whether non-engineer approvals reduce or increase engineering burden.

## 4. Standard Signal

**Money.** Standard Signal has YC seed financing and is opening to a limited number of accredited investors. It discloses no AUM, fund terms, audited performance, outside LPs, broker/custody setup, or regulatory registrations in the public material. It claims a live strategy with a Sharpe ratio above 3 and says performance is shared with qualified investors on request.

**Technology.** The user is a fund operator or qualified investor. Standard Signal's system ingests market data, news, filings, public-company information, macro signals, and possibly alternative data. Custom models research the world, form a trade thesis, decide position sizing and timing, execute through brokerage infrastructure, and log every decision for audit. Human operators should set risk limits, trading universe, maximum exposure, capital allocation, model-change approvals, and kill-switch conditions; the company explicitly says hard risk limits are enforced by infrastructure rather than the model.

The technical bet is reinforcement learning for trading judgment. A model that can summarize news is commodity; a model that learns when not to trade is the harder asset. The training loop needs clean historical data, execution-cost modeling, leakage prevention, walk-forward validation, market-impact estimates, drawdown controls, and out-of-sample regime testing. Live trading adds broker APIs, slippage, partial fills, data outages, monitoring, and compliance records.

The moat, if real, is proprietary trade outcome data, model-training loops, and full-stack infrastructure that links research, decision, execution, and audit. The risk is classic quant overfitting with modern language-model confidence. A short live track record can look excellent by variance, and public Sharpe claims without audited history are not enough. The system also faces capacity limits: the more capital follows a signal, the more the signal may decay.

**Founders.** Solo founder Michael Royzen previously co-founded Phind, trained LLMs that beat GPT-4 on HumanEval in 2024, and led a search/coding-model company to major usage. The ML experience is highly relevant; fund operations, compliance, and risk-management depth are less visible.

**Verdict — 8.0/10, Medium confidence.** Standard Signal has a bold, technically coherent thesis and a founder who can train models. Confidence depends on audited performance, controls, compliance, and evidence the live edge survives scaling.

**Sources:** [YC profile and launch, accessed June 24, 2026](https://www.ycombinator.com/companies/standard-signal) · [Company site, accessed June 24, 2026](https://standardsignal.com/)

## 5. Regbase

**Money.** Regbase has YC seed financing and no disclosed outside round, revenue, seat count, or retention. Its launch gives concrete client outcomes: finding $400 million in expiring school-safety grants for a protective-equipment client, identifying 80 greenhouse-gas disclosure requirements across 45 countries, and finding obscure non-English laws posted outside normal legal databases. These are useful evidence of workflow value, though customer names and paid status are not public.

**Technology.** A legal, compliance, policy, or government-vendor team gives Regbase a jurisdiction, topic, product category, or sales target. Regbase crawls and monitors official gazettes, draft bill portals, consultation pages, government websites, PDFs, local-language sources, and even social platforms where government bodies post updates. It extracts legal changes, translates and classifies them, links each finding to source evidence, and returns alerts, requirement lists, sales opportunities, or research summaries. Lawyers and researchers review relevance, legal interpretation, and client-facing advice.

The architecture needs source discovery, resilient crawling of poorly maintained websites, OCR, translation, entity resolution, jurisdiction taxonomy, legal-topic classification, deduplication, and change detection. The difficult part is provenance and recall. Missing one local rule can matter more than summarizing ten obvious updates well. A credible system needs source freshness, citation anchors, confidence scores, and human review queues for ambiguous legal effect.

Generic web search and LLM summaries are not enough because many sources are unindexed, non-English, hidden behind forms, or inconsistent across local governments. Regbase can build a moat through source coverage, corrected legal-topic labels, jurisdiction-specific monitors, and workflows that legal teams trust. Risks include unauthorized practice of law boundaries, false positives, false negatives, and the labor required to maintain source maps globally.

**Founders.** Natalie Aresta-Katz spent three years as a regulatory researcher at Paul, Weiss and previously built a commerce business that sold over $1 million of soap base on Amazon. Ethan Shea was a staff engineer at Vercel, built observability backend and a Vercel Agent, and worked at Citadel on a master tradable-instrument list. The founder mix combines real regulatory-research pain with systems engineering.

**Verdict — 7.9/10, Medium-high confidence.** Regbase has a narrow, valuable problem and strong evidence that ordinary search misses important regulatory changes. The open question is whether source coverage and legal accuracy scale without becoming a services firm.

## 6. Sherpa

**Money.** Sherpa has YC seed financing and no disclosed outside round, pricing, revenue, or customer count. Its YC profile says customers typically see a 15-20% lift in their primary conversion metric after 30 days. Its site names Lindy, Insforge, and Gauge as trusted users and presents a case where Sherpa designed, built, and ran 40 experiments in 30 days with a reported 30% total conversion lift. These are strong product signals, but methodology and baseline controls are not public.

**Technology.** A website owner installs Sherpa with a snippet or integration. Sherpa watches sessions, identifies friction points such as high-dwell/low-conversion sections, compares funnel behavior, and proposes new headlines, layouts, CTAs, animations, pricing presentation, onboarding flows, or landing pages. The human approves changes before they go live. Sherpa then runs A/B tests, monitors conversion lift, and deploys winners while discarding losers.

The system needs product analytics, session replay interpretation, DOM understanding, design generation, code or CMS modification, brand constraints, experiment assignment, statistical testing, QA, rollback, and performance monitoring. For agent-facing optimization, it likely needs to test what LLM browsers or shopping agents extract from the page and whether the site is recommended more often. The hard part is causal measurement. A generated variant can look better while seasonality, traffic source mix, or novelty effects explain the lift.

Commodity pieces include website analytics, A/B testing, LLM copy generation, and front-end code generation. Sherpa's moat could come from experiment history, visitor-behavior models, generated-design QA, and tight approval/deployment loops. Risks include site breakage, SEO impact, accessibility regressions, brand drift, privacy compliance, and false positive experiment results.

**Founders.** Ethan Kinnan was Lindy's first forward-deployed engineer and taught custom-agent building. Norbu Sonam built agent infrastructure at Lindy, ran 50-plus growth experiments there, tripled paywall conversion, and previously shipped image-generation products at Microsoft Copilot. This is strong founder-market fit for autonomous growth experiments.

**Verdict — 7.8/10, Medium-high confidence.** Sherpa has a measurable workflow and credible early customer examples. Confidence depends on statistically clean lift, safe deployment, and whether the product can optimize full funnels rather than surface-level pages.

## 7. Zolvo

**Money.** Zolvo has YC seed financing and no disclosed outside round. It says its first client paid an annual contract before the product existed, became an angel investor two weeks in, went live in less than a month, and replaced work that otherwise would have required a 14-person team. No contract value, named customer, loan volume, retention, or error-rate reduction is public.

**Technology.** A factoring or asset-based lending team connects bank accounts, loan management systems, funder monitoring systems, communications, invoices, and borrower records. Zolvo ingests payments, invoices, remittances, emails, SMS threads, and portfolio data. It matches messy wires to invoices, identifies exceptions, drafts or sends collections outreach, updates lender systems, and prepares monitoring/reporting for funders. Operators review exceptions, risky collections actions, borrower disputes, and any decision that changes funding or credit status.

The architecture has to reconcile financial state across systems that were not designed to agree. It needs bank-feed ingestion, invoice extraction, borrower/entity resolution, payment matching, collections workflow state, permissions, audit logs, and integrations into loan servicing systems. The hard engineering is correctness under messy data: partial payments, fees, chargebacks, duplicate invoices, changed bank references, disputed receivables, and borrower-specific rules.

Language models can draft messages and extract documents, but deterministic ledgers and reconciliation rules must own the money movement. A useful system should separate model suggestions from booked financial state, preserve every source document, and expose why a match or collection action was made. Zolvo's moat could come from lender-specific workflow templates, corrected reconciliation data, integrations, and becoming the operational layer for private-credit back offices. Risks are data security, financial-error liability, and implementation labor.

**Founders.** Isabela Rodriguez was the first go-to-market hire at Domu and says she helped close the company's first $1 million in revenue. Tony Montes was a founding AI engineer at Domu, built voice AI infrastructure at 100K-plus calls per day, and has AI research publications. The team has relevant fintech AI and early-enterprise sales experience.

**Verdict — 7.8/10, Medium confidence.** Zolvo has unusually concrete early customer behavior for a new vertical automation company. The rating depends on named retained lenders, reconciliation accuracy, implementation time, and liability controls.

## 8. Andco

**Money.** Andco has YC seed financing and no disclosed outside round. It says it is live with personal-injury firms across the United States, is exploring pilots with the world's largest injury law firms, and handles hundreds of cases and thousands of requests each month. It claims roughly eight hours saved in the first 24 hours of a case and more than 24 hours saved between intake and demand. Customer names, revenue, pricing, and retention are not public.

**Technology.** A personal-injury firm signs a new client and sends the case to Andco. The agents request police reports, open insurance claims with carriers, identify uninsured/underinsured motorist coverage, contact adjusters, request medical records and bills from providers, and follow up across fax, email, mail, web portals, SMS, and voice until documents arrive. Staff review retrieved records, exceptions, ambiguous coverage, failed requests, and downstream legal judgments.

This is an orchestration problem across obsolete channels. The stack needs case-state modeling, document request templates, provider and carrier directories, fax/voice/email/SMS infrastructure, portal automation, identity verification, deadlines, retry policies, and evidence capture. It must know which records are missing, what authority or release is required, and when a human should intervene. A generic legal summarizer is insufficient because value comes from obtaining the documents and maintaining a clean case timeline.

The hard engineering is reliable external execution. Providers and police departments use inconsistent forms, slow portals, mailed copies, and phone queues. Automation mistakes can delay a case or create compliance problems around health data. Andco needs HIPAA-aware controls, consent tracking, audit logs, call recording and summaries, provider-specific playbooks, and escalation thresholds. Defensibility could come from request-routing data, provider response patterns, and tight integration into PI workflows.

**Founders.** Ryan Xue previously co-founded Layup, was the first business hire at Origin, and worked at McKinsey and Google. David Lee was CTO at NoxuData, founding director of engineering at Seek AI, and worked at DataFox. Michael Lemm co-founded Layup and was an attorney at Kirkland & Ellis. The team combines legal, workflow, and startup execution experience.

**Verdict — 7.7/10, Medium-high confidence.** Andco is a practical agent company because the task is repetitive, channel-heavy, and measurable. The key proof is paid retention, retrieval accuracy, compliance, and whether automation scales beyond founder-supervised workflows.

## 9. Mount

**Money.** Mount has YC seed financing and no disclosed outside round, policy count, premium volume, carrier/reinsurance relationship, loss reserve, or claims history. It has announced the Mount AI Agent Liability Policy for mid-market and enterprise companies deploying autonomous AI agents, and it is hiring underwriting and GTM roles.

**Technology.** A company deploying production AI agents submits system details: model, prompts, tools, permissions, data access, human approvals, logs, and workflow boundaries. Mount scans the deployment, red-teams likely failure modes, quantifies operational risk, recommends mitigations, and underwrites residual liability coverage. Humans at the insured company review security findings and decide which mitigations to implement; Mount or its capacity partners decide what risk can be priced and insured.

The technical system must model agent behavior as an insurable process. It needs tool-permission graphs, data-classification, prompt-injection tests, unauthorized-action probes, workflow replay, incident severity scoring, and continuous monitoring when models or tools change. The insurance side needs underwriting rules, policy wording, exclusions, premium pricing, claims triage, fraud detection, and regulatory compliance. The strongest version of Mount is not just a scanner; it is a feedback loop between technical controls and realized losses.

The hard problem is calibration. Many AI-agent risks are new, low-frequency, and hard to price. Security tests can find weaknesses without predicting financial loss. A credible carrier needs production telemetry, incident definitions, actuarial assumptions, capacity, and clear policy language about what is covered. Commodity red-team tools and questionnaires are easy to copy. The moat would be proprietary loss data, underwriting models, insurance licenses/capacity, and procurement trust.

**Founders.** Fabian Amherd studied CS/ML at ETH Zurich and built real-time CNN systems for industrial robotics. John Bachmann comes from an insurance family and previously scaled an AI-first media company to $400K in annual revenue. The founder mix is promising, but senior insurance capacity and actuarial depth are not public.

**Verdict — 7.7/10, Medium confidence.** Mount addresses a real production-adoption blocker for AI agents. The missing proof is carrier structure, policy terms, pricing calibration, and claims-handling capability.

## 10. Oddpool

**Money.** Oddpool has YC seed financing and no disclosed outside round. The company says it launched a prosumer tier and grew to several thousand dollars in MRR within months, then moved upmarket toward quant funds and trading desks. It discloses no institutional contracts, data volume, paid retention, or exchange relationships.

**Technology.** A quant fund, prop shop, market maker, or hedge fund wants prediction-market data. Oddpool collects markets from multiple venues, maps similar contracts to a canonical identifier, normalizes wording, timestamps, order books, trades, prices, and deltas, and exposes search, analysis, and backtesting. The output is a unified data layer that a desk can query or feed into a strategy. Humans still review contract resolution rules, venue quality, legal availability, liquidity, and whether a backtest reflects tradable reality.

The hard technical problem is that event contracts are not ordinary tickers. Similar-looking markets can differ by resolution source, deadline, void rules, jurisdiction, settlement, fees, and liquidity. Oddpool needs universal symbology, tick-level history, full order-book depth, market metadata, venue adapters, corporate/action-like event handling, and careful records of rule changes. Backtests must model slippage, market impact, fees, resolution uncertainty, and unavailable historical liquidity.

Commodity pieces include market-data ingestion, databases, and dashboards. Oddpool's moat can come from high-quality normalization, clean historical depth, venue coverage, and the trust of serious desks. Risks are regulatory uncertainty, venue API fragility, thin liquidity, manipulation, and the possibility that prediction-market venues themselves build the preferred data layer.

**Founders.** Avi Arora has a master's in machine learning and says he trained a Microsoft prompt-injection model securing 20 billion requests per year. Ritesh Malpani has a master's in distributed systems and built Bloomberg buy-side trading infrastructure processing 100K-plus transactions per second. The pairing is directly relevant to market-data systems.

**Verdict — 7.5/10, Medium confidence.** Oddpool is a picks-and-shovels bet on prediction markets becoming institutional. The technical wedge is real, but proof requires institutional contracts, backtest quality, API reliability, and regulatory clarity.

**Sources:** [YC profile and launch, accessed June 24, 2026](https://www.ycombinator.com/companies/oddpool) · [Company site, accessed June 24, 2026](https://www.oddpool.com)
